# Управление зависимостями задач и групп задач в Airflow

**Зависимости (dependencies)** определяют порядок выполнения: какая задача должна достичь нужного состояния, чтобы запустилась следующая. В DAG задачи — узлы графа, зависимости — рёбра.

Термины:

- **Upstream task** — задача, которая должна достичь заданного состояния до запуска зависимой.
- **Downstream task** — зависимая задача, которая ждёт вышестоящую.

Ниже — зависимости внутри одного DAG. Зависимости между DAG: [Cross-DAG dependencies](https://www.astronomer.io/docs/learn/cross-dag-dependencies).

## Базовые зависимости

- Операторы сдвига: `t0 >> t1 >> t2 >> t3` или `t3 << t2 << t1 << t0`.
- Методы: `t0.set_downstream(t1)` и `t1.set_upstream(t0)`.

Несколько downstream от одной задачи: списки или кортежи:

```python
t0 >> t1 >> [t2, t3]
# или
t0 >> t1 >> (t2, t3)
```

Зависимость «список от списка» так задать нельзя (`[t0, t1] >> [t2, t3]` — ошибка). Для связей между списками используются функции зависимостей.

## Функции зависимостей

- **chain** — последовательность и параллельные «слои» одной длины: `chain(t0, t1, [t2, t3, t4], [t5, t6, t7], t8)`. Соседние списки/кортежи должны быть одной длины.
- **chain_linear** — каждый элемент нижестоящего списка зависит от каждого элемента вышестоящего: `chain_linear([t0, t1], [t2, t3, t4])`.

Удобны для задач, созданных в цикле и собранных в список.

## Динамически маппленные задачи

Зависимости задаются так же, как для обычных задач: `start >> multiply_obj >> end`. При правиле по умолчанию `all_success` все экземпляры маппленной задачи должны успешно завершиться, чтобы запустилась downstream.

## Зависимости с Task Groups

Зависимости внутри группы задаются в контексте группы (`t1 >> t2`). Связь группы с остальным DAG — снаружи: `t0 >> tg1() >> t3`. Зависимости между группами, между задачами из разных групп и между группами и отдельными задачами задаются через те же операторы и функции.

## TaskFlow API

При использовании `@task` зависимости часто выводятся из аргументов: результат вышестоящей задачи передаётся в нижестоящую: `print_the_cat_fact(get_a_cat_fact())`. Можно сохранить вызов в переменную и передать в несколько задач: `num = get_num(); add_one(num); add_two(num)`.

Смешение декораторов и традиционных операторов: вызов декорированной задачи сохранить в переменную и использовать в цепочке: `_my_task_1 = my_task_1(); chain(_my_task_1, my_task_2)`.

## Trigger rules (правила срабатывания)

По умолчанию задача запускается, когда **все** непосредственно вышестоящие задачи успешны. Это меняется параметром **trigger_rule**. Варианты: `all_success`, `all_failed`, `all_done`, `all_skipped`, `one_failed`, `one_success`, `one_done`, `none_failed`, `none_failed_min_one_success`, `none_skipped`, `always`. Подробнее: [Trigger rules](trigger-rules.md).
