# Управление кодом Airflow

В Airflow пайплайны описываются кодом, поэтому к ним применимы практики разработки: версионирование и CI/CD. По мере роста использования Airflow важно организовать код так, чтобы его было удобно поддерживать.

Здесь под **проектом** имеется в виду набор DAG и вспомогательных файлов, развёрнутых в одно окружение (deployment). У разных команд могут быть свои проекты и свои деплойменты.

## Структура проекта

Рекомендуется один каталог и один репозиторий на проект. Пример структуры (Astronomer):

```
.
├── dags/              # DAG-файлы
│   ├── example-dag.py
│   └── redshift_transforms.py
├── Dockerfile
├── include/           # скрипты, SQL и т.д.
│   └── sql/
│       └── transforms.sql
├── packages.txt       # системные пакеты
├── plugins/           # кастомные/community плагины
│   └── example-plugin.py
└── requirements.txt  # Python-зависимости
```

Создать заготовку можно командой `astro dev init` (Astro CLI). Без Docker структура может отличаться; главное — выбрать соглашения и придерживаться их.

## Когда выносить в отдельные проекты

Обычно один репозиторий на один деплоймент. Отдельные проекты (и отдельные деплойменты) имеют смысл, если:

- **Доступ:** RBAC нужен на уровне деплоймента. Если команда A не должна видеть DAG команды B — два проекта и два деплоймента.
- **Инфраструктура:** одни DAG лучше подходят Kubernetes executor, другие — Celery; можно разнести по разным проектам с разной инфраструктурой.
- **Зависимости:** конфликтующие Python/OS зависимости проще изолировать разными проектами.

Реже DAG из нескольких репозиториев собирают в один деплоймент — тогда объединение делают через CI/CD (для Astro Private Cloud — NFS или Git-Sync).

## Переиспользование кода

Общий код (кастомные хуки, операторы, шаблоны DAG) лучше хранить в отдельном репозитории и подключать в проекты (через зависимости, приватные пакеты, CI/CD). Так изменения вносятся в одном месте. Подробнее: [Sharing code across projects](https://www.astronomer.io/docs/learn/sharing-code-multiple-projects).

## Поток разработки

Ту же структуру можно использовать для продвижения кода по средам dev → QA → production. Типичный подход — ветки: `dev`, `qa`, `main`; `main` соответствует продакшену. CI/CD автоматизирует деплой из нужной ветки. Альтернатива — feature-ветки; выбор зависит от процессов организации.
