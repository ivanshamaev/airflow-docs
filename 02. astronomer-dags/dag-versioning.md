# Версионирование DAG и DAG bundles

## Версии DAG

При каждом изменении структуры DAG (добавление/удаление задач, изменение зависимостей) Airflow создаёт новую **версию** DAG. Каждый DAG run привязан к версии — так можно понимать, какой именно код выполнялся в прошлом запуске. Версии отображаются в UI (например, в Graph view можно выбрать версию) и в API.

## DAG bundle (каталог DAG)

**DAG bundle** — каталог (или список каталогов), из которого Airflow загружает DAG. Задаётся конфигурацией (например, `dags_folder`). Airflow периодически сканирует файлы в bundle, выполняет Python-файлы и регистрирует обнаруженные объекты DAG.

- По умолчанию учитываются только файлы, в которых встречаются подстроки `airflow` и `dag` (без учёта регистра). Отключить эту эвристику можно через **DAG_DISCOVERY_SAFE_MODE**.
- Файл **.airflowignore** в каталоге DAG (или в подкаталоге) задаёт шаблоны имён файлов, которые нужно игнорировать (синтаксис как в .gitignore). Это уменьшает парсинг лишних файлов.

## Несколько DAG bundles

В некоторых развёртываниях можно настроить несколько путей загрузки DAG (несколько bundle), чтобы подтягивать DAG из разных источников (разные репозитории, окружения). Подробности зависят от способа деплоя (Kubernetes, Docker, Astro и т.д.). См. [DAG versioning and DAG bundles](https://www.astronomer.io/docs/learn/airflow-dag-versioning).

## Рекомендации

- Храните DAG в Git; деплой через CI/CD сохраняет историю и версии кода.
- Используйте .airflowignore, чтобы не парсить тестовые скрипты и лишние модули в папке DAG.
