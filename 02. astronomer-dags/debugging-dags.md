# Отладка DAG в Airflow

## Проверка парсинга

- **airflow dags list** — список DAG; при ошибках импорта они отображаются в **airflow dags list-import-errors**.
- Исправляйте синтаксические ошибки и ошибки импорта до деплоя: парсер запускается регулярно, ошибки видны в логах планировщика и в UI.

## Локальный запуск задачи

- **airflow tasks test \<dag_id\> \<task_id\> \<date\>** — выполнение одной задачи локально без записи в метаданные как в продакшене. Удобно для проверки логики и доступа к контексту/XCom.
- В тестах можно подставлять нужную дату и параметры, чтобы проверить шаблоны и ветвление.

## Логи и XCom

- В UI: открыть DAG run → задача → вкладка **Logs**. Логи показывают stdout/stderr задачи и сообщения планировщика.
- **XCom**: вкладка XCom у Task Instance или Admin → XCom — проверка, какие значения передаются между задачами. Для больших значений убедитесь, что используете XCom в рамках ограничений или промежуточное хранилище.

## Шаблоны

- Убедитесь, что подставляемые переменные доступны в контексте для данного типа запуска (scheduled vs manual vs asset-triggered). При ручном запуске с `logical_date=None` части ключей контекста (ds, ts, data_interval_*) нет — обращение к ним вызовет KeyError. См. [Контекст](airflow-context.md), [Jinja](jinja-templating.md).

## Перед деплоем

- Прогоняйте линтеры и тесты в CI; проверяйте список import errors после деплоя.
- Используйте [лучшие практики](dag-best-practices.md): не тянуть тяжёлую логику на верхний уровень, ограничивать размер XCom, задавать таймауты и retries.

Подробнее: [Debugging DAGs](https://www.astronomer.io/docs/learn/debugging-dags).
