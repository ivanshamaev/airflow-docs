# Задачи (Tasks)

**Задача (Task)** — базовая единица выполнения в Airflow. Задачи объединяются в [DAG](dags.md), между ними задаются зависимости «выше по потоку» (upstream) и «ниже по потоку» (downstream), что определяет порядок выполнения.

Есть три основных вида задач:

1. **Операторы (Operators)** — готовые шаблоны задач, из которых собирают большую часть DAG.
2. **Сенсоры (Sensors)** — подкласс операторов, которые ждут наступления внешнего события.
3. **TaskFlow** — функция Python, обёрнутая декоратором `@task` и упакованная в задачу.

Внутри все они являются подклассами `BaseOperator`; понятия «задача» и «оператор» часто используют как синонимы, но удобно различать: операторы и сенсоры — шаблоны, а при вызове в файле DAG вы создаёте задачу.

## Зависимости между задачами

Главное при работе с задачами — задать их связи. Сначала объявляют задачи, затем зависимости.

> **Терминология:** вышестоящая (upstream) задача — та, что непосредственно предшествует другой; нижестоящая (downstream) — непосредственно следующая. Раньше использовали «родитель/потомок», но это не иерархия в смысле «все предки», а только прямая связь.

Зависимости задают операторами `>>` и `<<`:

```python
first_task >> second_task >> [third_task, fourth_task]
```

или методами `set_upstream` / `set_downstream`:

```python
first_task.set_downstream(second_task)
third_task.set_upstream(second_task)
```

Оба варианта эквивалентны; рекомендуется использовать `>>` и `<<` как более наглядные.

По умолчанию задача запускается, когда все её вышестоящие задачи успешно завершились. Поведение можно менять (ветвление, ожидание части предков, учёт истории запусков) — см. [Control Flow](dags.md) в документации по DAG.

Задачи по умолчанию не передают друг другу данные и выполняются независимо. Для передачи данных между задачами используется [XComs](https://airflow.apache.org/docs/apache-airflow/stable/core-concepts/xcoms.html).

## Экземпляры задач (Task Instances)

Так же, как DAG при каждом запуске создаёт **Dag Run**, каждая задача при запуске DAG создаёт **Task Instance** — конкретный запуск этой задачи для данного DAG (и интервала данных). У экземпляра задачи есть состояние.

Возможные состояния Task Instance:

| Состояние | Описание |
|-----------|----------|
| `none` | Задача ещё не поставлена в очередь (зависимости не выполнены) |
| `scheduled` | Планировщик решил, что зависимости выполнены и задачу можно запускать |
| `queued` | Задача передана исполнителю и ждёт воркера |
| `running` | Задача выполняется на воркере |
| `success` | Задача завершилась без ошибок |
| `restarting` | Запрошен перезапуск во время выполнения |
| `failed` | Ошибка при выполнении |
| `skipped` | Пропущена из-за ветвления, LatestOnly и т.п. |
| `upstream_failed` | Вышестоящая задача упала и по правилу срабатывания текущая не выполняется |
| `up_for_retry` | Задача упала, но остались попытки — будет перезапущена |
| `up_for_reschedule` | Сенсор в режиме `reschedule` |
| `deferred` | Выполнение отложено (defer) на триггер |
| `removed` | Задача удалена из DAG после начала запуска |

В норме задача проходит: `none` → `scheduled` → `queued` → `running` → `success`.

### Терминология связей

У экземпляра задачи два типа связей:

- **upstream/downstream** — другие задачи в том же DAG Run (тот же интервал данных).
- **previous/next** — экземпляры той же задачи в других запусках того же DAG (другие интервалы). Это не «выше/ниже по потоку».

> В старой документации «previous» иногда означало «upstream»; если встретите — это устаревшее употребление.

## Таймауты

- **execution_timeout** — максимальное время одного выполнения задачи (для всех задач, включая сенсоры). Задаётся как `datetime.timedelta`. При превышении вызывается `AirflowTaskTimeout`.
- У сенсоров в режиме `reschedule` есть также **timeout** — максимальное время с начала первого запуска до успешного срабатывания. При превышении — `AirflowSensorTimeout`, сенсор падает без повторных попыток.

## SLA

Функция SLA из Airflow 2 в 3.0 удалена; в 3.1 заменена на [Deadlines Alerts](https://airflow.apache.org/docs/apache-airflow/stable/howto/deadline-alerts.html).

## Специальные исключения

В коде задачи можно явно задать состояние:

- **AirflowSkipException** — пометить задачу как пропущенную (skipped).
- **AirflowFailException** — пометить как failed и не использовать оставшиеся попытки.

Полезно, когда задача сама понимает контекст: например, пропуск при отсутствии данных или быстрый fail при невалидном API-ключе.

## Таймаут heartbeat экземпляра задачи

Экземпляр задачи может «зависнуть» в состоянии `running` (например, воркер убит по OOM). Такие задачи раньше называли zombie. Airflow периодически их находит, помечает как failed или перезапускает при наличии попыток. Heartbeat может истечь из-за OOM воркера, рестарта воркера (Kubernetes liveness), переноса воркера на другой узел и т.д.

## Конфигурация исполнителя (Executor)

Некоторые [исполнители](https://airflow.apache.org/docs/apache-airflow/stable/core-concepts/executor/index.html) (например, KubernetesExecutor) позволяют задавать параметры на уровне задачи через аргумент **executor_config**:

```python
MyOperator(...,
    executor_config={
        "KubernetesExecutor": {"image": "myCustomDockerImage"}
    }
)
```

Допустимые ключи зависят от исполнителя — см. документацию по конкретному executor.
