# Что такое Apache Airflow®?

[Apache Airflow®](https://github.com/apache/airflow) — открытая платформа для разработки, планирования и мониторинга пакетных (batch) рабочих процессов. Расширяемый Python-фреймворк Airflow позволяет строить пайплайны, интегрирующиеся практически с любой технологией. Веб-интерфейс помогает визуализировать, управлять и отлаживать процессы. Airflow можно запускать в разных конфигурациях — от одного процесса на ноутбуке до распределённой системы для больших нагрузок.

## Рабочие процессы как код

Рабочие процессы в Airflow описываются целиком на Python. Такой подход «workflows as code» даёт ряд преимуществ:

- **Динамичность:** пайплайны задаются в коде, что позволяет динамически генерировать и параметризовать DAG.
- **Расширяемость:** во фреймворк входит множество встроенных операторов, и его можно расширять под свои задачи.
- **Гибкость:** Airflow использует шаблонизатор [Jinja](https://jinja.palletsprojects.com/) для тонкой настройки.

## DAG (Dag)

**Dag** — модель, в которой собрано всё необходимое для выполнения рабочего процесса. В число атрибутов DAG входят:

- **Расписание (Schedule):** когда должен запускаться процесс.
- **Задачи (Tasks):** [задачи](core-concepts/tasks.md) — отдельные единицы работы, выполняемые на воркерах.
- **Зависимости задач:** порядок и условия выполнения [задач](core-concepts/tasks.md).
- **Обратные вызовы (Callbacks):** действия по завершении всего процесса.
- **Дополнительные параметры** и другие детали эксплуатации.

Пример простого DAG:

```python
from datetime import datetime

from airflow.sdk import DAG, task
from airflow.providers.standard.operators.bash import BashOperator

# DAG — это рабочий процесс, набор задач
with DAG(dag_id="demo", start_date=datetime(2022, 1, 1), schedule="0 0 * * *") as dag:
    # Задачи задаются операторами
    hello = BashOperator(task_id="hello", bash_command="echo hello")

    @task()
    def airflow():
        print("airflow")

    # Зависимости между задачами
    hello >> airflow()
```

В примере:

- DAG с именем `"demo"`, ежедневный запуск с 1 января 2022 года. DAG в Airflow — это и есть рабочий процесс.
- Две задачи: одна через `BashOperator` (команда оболочки), вторая — функция Python с декоратором `@task`.
- Оператор `>>` задаёт зависимость и порядок выполнения.

Airflow разбирает скрипт, планирует и выполняет задачи в заданном порядке. Статус DAG `"demo"` отображается в веб-интерфейсе.

Задачи могут выполнять практически любой код: задачу Spark, копирование файлов между хранилищами, отправку уведомления и т.д. В сетке (Grid) каждый столбец — один запуск DAG; кроме графа и сетки есть и другие представления (например, Dag Overview).

> **Примечание.** Термин «DAG» восходит к понятию «ориентированный ациклический граф», но в Airflow он давно означает больше, чем просто структура данных. Поэтому в документации используется написание **Dag**.

## Зачем нужен Airflow®?

Airflow — платформа оркестрации пакетных процессов. Гибкий фреймворк, много встроенных операторов и простая интеграция с новыми технологиями.

Если у процессов есть явное начало и конец и они запускаются по расписанию — они хорошо ложатся на DAG в Airflow.

Если удобнее кодировать, чем нажимать кнопки — Airflow как раз про это. Описание процессов кодом даёт:

- **Версионирование:** история изменений, откат, совместная работа.
- **Командная работа:** несколько разработчиков могут вести один код пайплайнов.
- **Тестирование:** проверка логики юнит- и интеграционными тестами.
- **Расширяемость:** использование существующих компонентов или своих.

Гибкие правила планирования и выполнения позволяют описывать сложные повторяющиеся пайплайны. Из веб-интерфейса можно вручную запускать DAG, смотреть логи и статусы задач, делать backfill по историческим данным или перезапускать только упавшие задачи.

Платформа сильно настраивается: через [Public Interface for Airflow 3.0+](https://airflow.apache.org/docs/apache-airflow/stable/public-airflow-interface.html) можно расширять и менять почти любую часть системы — от операторов до плагинов UI и логики выполнения.

Airflow — open source: вы опираетесь на компоненты, которые разрабатывает и поддерживает сообщество. Много материалов: блоги, книги, доклады; общение — через [сообщество](https://airflow.apache.org/community), [Slack](https://s.apache.org/airflow-slack) и рассылки.

## Когда Airflow не лучший выбор

Airflow® рассчитан на **конечные пакетные** процессы. Запуск по CLI или REST API возможен, но Airflow **не предназначен** для постоянно работающих, событийных или стриминговых нагрузок. При этом он часто дополняет стриминговые системы (например, Apache Kafka): Kafka пишет данные в хранилище в реальном времени, а Airflow периодически забирает и обрабатывает их пачками.

Если удобнее управлять процессами через интерфейс, а не код — Airflow может не подойти. Веб-UI упрощает управление, но описание процессов кодом остаётся основой, так что некоторое программирование всё равно нужно.
